# 股票分析软件 - 公式生成与运行分析 PRD

## 1. 产品概述

### 1.1 产品定位
股票分析软件是一款基于技术分析规则的智能选股与交易信号系统。系统通过多维度技术指标规则库，结合动态权重优化算法，为用户生成个性化的买入/卖出交易信号。

### 1.2 核心价值
- **自适应权重**：根据个股历史数据自动优化各技术规则的权重，不同股票适用不同参数组合
- **多规则协同**：14条买入规则 + 15条卖出规则 + 1条必卖规则 + 2条买入限制规则，多规则交叉验证降低误判
- **防假突破机制**：买入信号需阳线确认 + 量能确认，有效过滤假突破
- **强制止损**：均线趋势打破必卖规则，确保上升趋势被破坏时及时离场

---

## 2. 功能架构

### 2.1 整体流程

```
用户添加股票 → 抓取历史数据 → 计算权重(公式生成) → 运行分析 → 查看信号
```

### 2.2 功能模块关系

| 模块 | 前置依赖 | 说明 |
|------|----------|------|
| 股票管理 | 无 | 添加/删除股票 |
| 数据抓取 | 股票管理 | 抓取日/周/月/分钟K线数据 |
| 公式计算(权重) | 数据抓取(≥200条日线) | 基于历史数据优化各规则权重 |

| 信号展示 | 运行分析 | 在K线图上标注B/S信号 |

## 3. 公式生成（权重计算）

### 3.1 功能描述
基于个股历史K线数据，自动计算每条技术规则的权重，形成该股票专属的"分析公式"。不同股票因走势特征不同，各规则权重不同。

### 3.2 入口与触发
- **前端入口**：「公式计算」Tab → 点击「计算权重」按钮
- **API接口**：`POST /api/calculate_weights`
- **请求参数**：`{ "stock_code": "000001" }`

### 3.3 前置条件
- 股票已有日线K线数据
- 日线数据量 ≥ 200条（不足时系统自动尝试补充至800条）

### 3.4 核心算法流程

#### 3.4.1 规则信号预计算
对每条规则，遍历历史数据每一天，记录规则是否触发，生成布尔信号表：
```
规则名 → [False, False, True, False, True, ...]  (长度=数据天数)
```

#### 3.4.2 质量评分预计算
对每个触发点，计算信号质量评分（0~1），综合以下因素：
- **规则共振度**（45%）：当天触发的买入/卖出规则数量占比
- **动量方向**（35%）：近5日价格动量与信号方向一致性
- **基础分**（20%）：触发即得

#### 3.4.3 自适应回测
对每条规则进行自适应回测，统计：
- 胜率（win_rate）
- 平均收益率（avg_return）
- 触发次数（count）
- 盈亏比（profit_factor）
- 平均持有天数（avg_hold）
- 最大回撤（avg_max_drawdown）

回测参数：
| 参数 | 默认值 | 说明 |
|------|--------|------|
| min_hold | 3天 | 最短持有期 |
| max_hold | 20天 | 最长持有期 |
| stop_loss_pct | 5% | 止损线 |
| take_profit_pct | 10% | 止盈线 |

#### 3.4.4 深度搜索优化
遍历不同深度参数（step=10, max_depth=200），寻找最优深度：
1. 在给定深度下，找出历史数据中的价格极值点（高点/低点）
2. 计算各规则在极值点附近（window=5天）的加权质量得分
3. 加入时间衰减因子（decay_factor=3.0），越接近极值点的信号权重越高
4. 归一化得到各规则的原始权重

#### 3.4.5 梯度惩罚机制
根据回测表现对权重进行惩罚调整：

| 指标 | 条件 | 惩罚系数 |
|------|------|----------|
| 样本量 | count < 3 | ×0.3 |
| 胜率 | ≥65% | ×1.0 |
| 胜率 | [55%, 65%) | ×0.92 |
| 胜率 | [45%, 55%) | ×0.80 |
| 胜率 | [35%, 45%) | ×0.60 |
| 胜率 | <35% | ×0.35 |
| 均收益 | ≥3% | ×1.0 |
| 均收益 | [1%, 3%) | ×0.95 |
| 均收益 | [0%, 1%) | ×0.85 |
| 均收益 | <0% | ×0.50 |
| 盈亏比 | ≥1.5 | ×1.0 |
| 盈亏比 | [1.0, 1.5) | ×0.95 |
| 盈亏比 | [0.7, 1.0) | ×0.80 |
| 盈亏比 | <0.7 | ×0.45 |

最终权重 = 原始权重 × 胜率惩罚 × 收益惩罚 × 盈亏比惩罚

#### 3.4.6 规则层级分类
根据最终权重与最大权重的比值分类：

| 层级 | 比值范围 | 含义 |
|------|----------|------|
| 核心 | ≥80% | 最重要规则，主导信号方向 |
| 重要 | [50%, 80%) | 重要辅助规则 |
| 辅助 | [20%, 50%) | 补充验证规则 |
| 弱 | <20% | 影响较小，可能被惩罚 |

## 4. 运行分析

### 4.1 功能描述
使用已计算的权重，对历史K线数据逐日扫描，生成买入(B)/卖出(S)信号，并配对计算收益率。

### 4.2 入口与触发
- **前端入口**：「智能分析」Tab → 点击「运行分析」按钮
- **API接口**：`GET /api/analyze?stock_code=000001`
- **前置条件**：必须先完成「计算权重」

### 4.3 核心算法流程

#### 4.3.1 信号生成主循环
从第60个交易日开始，逐日计算：

**Step 1: 买入确认检查**
如果前一天有待确认的买入候选（pending_buy）：
- 检查当日是否为阳线：`close[idx] > open[idx]` 且 `close[idx] > close[prev_idx]`
- 满足 → 确认B信号
- 不满足 → 作废该候选

**Step 2: 计算买卖得分**
```
buy_score = Σ(触发的买入规则权重)
sell_score = Σ(触发的卖出规则权重)
```

**Step 3: 判断必卖条件**
- 检查必卖规则是否触发（均线趋势打破必卖）
- 必卖触发且当前持仓 → 强制卖出，confidence=1.0

**Step 4: 判断计数卖出条件**
- 触发的卖出规则数 ≥ 2（sell_trigger_count=2）
- 且当前持仓
- 且至少1条有权重的卖出规则触发
- 且 sell_score > buy_score

**Step 5: 判断买入信号**
- buy_score > sell_score 且 buy_score > 0 → 买入候选

**Step 6: 买入限制过滤**
- 触发「未突破5日线不买」或「阴线不买」→ 取消买入信号

**Step 7: 趋势过滤**
- 买入信号在下跌趋势中（MA20下降且价格<MA20）→ 取消

**Step 8: 最低分数过滤**
- 信号的主导得分 < 0.05（min_score_threshold）→ 取消

**Step 9: 触发质量过滤**
- 加权胜率 < 50% 且 加权盈亏比 < 1.0 → 取消

**Step 10: 信号间隔过滤**
- 与上一个同方向信号间隔 < 3天 → 取消
- 与上一个信号同方向 → 取消

**Step 11: 置信度过滤**
- confidence < 0.55（medium阈值）→ 不生成信号

**Step 12: 买入信号暂存**
- 买入候选暂存到 pending_buy，等待下一天确认
- 卖出信号直接生成

#### 4.3.2 信号配对
按时间顺序，将B信号与后续最近的S信号配对，计算：
- 买入日期/价格、卖出日期/价格
- 收益率 = (卖出价 - 买入价) / 买入价 × 100%

#### 4.3.3 统计汇总
- 总信号数、买入信号数、卖出信号数
- B-S配对数
- 活跃买入/卖出规则列表
- 买卖权重字典

## 6. 买入信号确认机制

### 6.1 两日确认规则
买入信号不立即生效，需经过以下确认流程：

```
Day N: 买入规则触发 → 生成买入候选(pending_buy)
Day N+1: 确认检查
  ├── 阳线确认: close[N+1] > open[N+1] 且 close[N+1] > close[N]  → 通过
  ├── 通过 → 在Day N+1生成B信号
  └── 不通过 → 候选作废，不生成信号
```

### 6.2 设计目的
- 防止假突破：单日突破后第二天回落的情况被过滤

## 7. 卖出信号触发机制

### 7.1 三级卖出优先级

| 优先级 | 类型 | 触发条件 | 置信度 |
|--------|------|----------|--------|
| 1（最高） | 必卖 | 必卖规则触发 + 当前持仓 | 1.0 |
| 2 | 计数卖出 | ≥2条卖出规则触发 + 持仓 + sell>buy | sell_score/(buy+sell) |

### 7.2 卖出过滤条件
- 信号间隔<3天的卖出被过滤（必卖/计数卖出除外）

## 8. 统一公式生成（旧版兼容）

### 8.1 功能描述
除动态权重引擎外，系统还保留了基于K线形态识别+DTW模板匹配的统一公式生成功能。

### 8.2 API接口
- 生成公式：`POST /api/generate_formula`
- 查询公式状态：`GET /api/formula_status?stock_code=000001`

### 8.3 生成流程
1. 遍历61种TA-Lib K线形态，统计每种形态在历史数据中的出现次数和平均收益率
2. 筛选出现次数≥3的形态，按平均收益率排序，取Top10看涨/看跌形态
3. 使用DTW模板匹配算法，计算V型底/W底/倒V顶/M头的动态阈值
4. 生成公式文档，存入MongoDB，有效期30天

### 8.4 DTW模板定义
| 模板名 | 方向 | 归一化序列 |
|--------|------|-----------|
| V | 底部 | [-1, -0.5, 0, 0.5, 1] |
| W | 底部 | [-1, 0, -0.5, 0, 0.5, 1] |
| INVERSE_V | 顶部 | [1, 0.5, 0, -0.5, -1] |
| M_TOP | 顶部 | [1, 0.3, 0.8, 0.2, 0.6, 0] |

  - 卖出规则卡片列表（绿色边框），按权重降序排列
  - 每张卡片包含：规则名、层级标签（核心/重要/辅助/弱，颜色区分）、权重百分比、原始权重、胜率、均收益、盈亏比、均持有天数、触发次数、惩罚信息

### 10.2 智能分析Tab
- **运行分析按钮**：点击后调用 `/api/analyze`，需先完成权重计算
- **今日建议**：买入(红)/卖出(绿)/观望(灰) 徽章
- **综合评分**：高(红)/中(黄)/低(灰) 评分徽章
- **B/S分数**：分别显示买入得分和卖出得分
- **触发规则标签**：以彩色标签展示当日触发的规则
- **信号原因列表**：详细列出信号触发原因
- **历史信号表格**：最近25条信号，含类型、日期、收盘价、评分、收益率、原因
- **K线图信号标注**：在ECharts K线图上标注B/S信号点

### 10.3 状态管理
- 切换股票时清除权重缓存和结果展示
- 权重计算成功后标记 `weightsCalculated = true`
- 运行分析依赖权重计算完成

## 11. 技术实现

### 11.1 后端架构
| 文件 | 职责 |
|------|------|
| app.py | Flask应用入口，静态文件服务 |
| routes.py | 股票CRUD、数据抓取、K线/财务数据API |
| unified_routes.py | 公式计算、运行分析、公式状态API |
| dynamic_weight_engine.py | 动态权重引擎核心算法 |
| unified_scanner.py | 统一信号扫描器（旧版+动态权重） |
| unified_rule_generator.py | 统一公式生成器（K线形态+DTW） |
| signal_dtw.py | DTW底部模板匹配检测器 |
| sell_dtw.py | DTW顶部模板匹配检测器 |
| signal_rules.py | 技术指标信号规则（旧版） |
| models.py | MongoDB数据模型 |
| db.py | MongoDB连接 |
| scraper.py | 东方财富数据抓取 |
| unified_config.yaml | 统一引擎配置文件 |

### 11.2 并发优化
- 规则信号预计算使用 `ThreadPoolExecutor` 并行（max_workers=min(规则数, 6)）
- 回测统计使用并行计算
- 极值点标签统计支持并行

### 11.3 缓存机制
- 权重结果缓存在内存 `_weight_cache` 中，避免重复计算
- 统一公式缓存在MongoDB中，有效期30天
- 配置文件带mtime检查的内存缓存

## 12. 异常处理

| 场景 | 处理方式 |
|------|----------|
| 数据不足（<200条） | 自动尝试补充至800条，仍不足则返回错误提示 |
| 股票代码无效 | 添加时校验，返回错误 |
| 规则计算异常 | 单条规则异常不影响其他规则，异常规则返回False |
| 网络超时 | 前端显示"网络错误"Toast |
| 重复抓取 | 检测抓取状态，防止并发抓取 |
| 公式过期 | 统一公式30天有效期，过期后重新生成 |
